总的来说，MVCC对于MySQL支持高并发的事务起了很大帮助，不是单纯的借助行级锁来实现事务。每行记录会有一些隐藏的属性，一般有事务号，undo指针等信息。MVCC并不是完整意义上的多版本控制，而是类似于乐观锁的机制，在比对之前的数据上，如果能覆盖操作就执行覆盖事务完成，不成功就会滚事务。

这个时候得到提醒

我们找到了MySQL的死锁日志,该锁产生是由于事务A要删除a = 3的记录，但是他已经持有了a=24这条记录的X锁，事务B在已经持有a=3这条记录的X锁之后，需要获得a=24的这把锁。
innodb侦测到了死锁的发生会滚了事务B，这是由于两个session之间发生的事务，对应到特性的语言上就是两个线程之间发生了死锁。

Redis的底层数据结构以及原理

2018-11-05

写在前面

Redis是个key-value的数据库，有很好的读写性能，有丰富的数据结构，在业务场景上也有很多优势。

Redis的数据结构

  • 动态字符串SDS,(Simple Dynamic String), 相比与C的字符串，SDS有自身的优势, 下面是用Java给出的SDS的类型，具体Redis底层是用C语言编写的。

SDS的数据类

  1. SDS空间预分配 & 惰性空间释放

SDS的空间与分配很有意思，预分配体现在预先这个词语上，我们都知道C语言扩展空间占用实际上是很费资源的，SDS的空间预分配用伪代码可以表示为：

从伪代码看出并不是每次扩展数据都需要alloc空间的，如果N次扩展，SDS的空间预分配策略是最大申请N次。

惰性空间释放，意味着当SDS的实际数据缩短时，空间并没有回收，只是把可用的size放在了free里面，这样下次再申请空间的时候，就不用再去acllocSize了。
当然这样会造成空间的浪费和内存碎片的产生，SDS提供了API来强制回收这部分的空间。

  • 链表

列表键就是双向链表来维护的数据，便于插入和删除数据. (lrange的执行效率？以及具体的执行方式)
使用的对象和场景: list

  • 字典

字典顾名思义就是以及对照表，通过查询字典能找到对应关系，然后找到对应的键值信息。对应的数据结构: hash表，使用链表发来解决冲突。使用的对象和场景: 数据库

  1. rehash（扩容与收缩）, 渐进式异步rehash
    当字典的容量不足的时候需要扩容，其实字典的hash维护了两个数组链表，h0和h1，当h1的容量达到阈值时，需要重新维持正常的hash负载因子，这个时候需要扩容重新rehash，类似于hashMap的rehash
    但是，redis的hash过程是渐进式的，当准备迁移数据到h1的时候，h0和h1都启用，此时的hash表是双读的，但是写只在新的数组链表写，随着数据的迁移，h0的数据不断的迁移到h1,直到h0数据为空。
    Redis的hash表维护了一个迁移的rehashIndex，因为迁移数据是常态，在高写入的情况下，系统是不迁移数据的。而且迁移数据是一个异步的操作，不影响读写。
  • 跳跃表

使用的对象和场景: zset, 集群的节点的内部结构

跳跃表的是有序集合的底层实现之一，跳跃表分为，跳跃表信息 + 跳跃节点（N个数据节点）

  • 整数集合

使用场景：是集合键的底层实现方式之一，如果ZSet的元素数量不多，那Redis就会使用整数集合来存储这个集合的键
数据结构：
扩容（有可能改变整个集合的数据类型，从int_8 升级到int_16/int_21/int_64）和缩容

不同数据结构的编码对象

Redis不同对象是用的底层数据结构

  • int
  • 字符串（embStr, raw）
  • list列表对象（linkedList, zipList, quickList）

  • hash对象(HSET)（ziplist, hashTable）, 在hash的key数量数量较少的时候，或者某个key的长度较大的时候会引起编码的改变，从ziplist -> hashTable
    Redis底层的存储是较小的时候index serach的时候是O(N)

  • 集合对象(SET),编码方式分为intSet/hashTable,intSet涉及全部值都是int的整型以及数量在512字节一下（有setting可以设置），只要超过这字节数量
    或者键的值里面有字符串出现，那么底层的intSet编码就会变成hashTable。特别的，如果采用hashTable的编码就会用键的值来作为hashTable的key，值为NULL
    问题：（如果用hashtable来作为底层的实现，hashTable是有底层的扩容机制的和采用链表法处理hash冲突，那现在Set的值如果很大的时候，每个值都是hash表的一个键，
    会引起极大的浪费？）

  • 有序集合对象（ZSET）
    有序集合对象在业务场景中会用到很多，排行榜等相关需求都需要用到ZSET的数据结构，那底层的ZSET是怎么实现的呢？

底层的数据结构是用两种方式来实现的，

  1. ziplist, 就是压缩列表，插入的时候用两个相邻的节点来保存数据，前面的节点保存数据的值，后面的节点保存数据的SCORE
  2. 字典 + 跳跃表来实现的，下面这张图很好的说明了ZSET的数据结构, 如果用字典和跳跃表的结构来实现的时候，实际上底层数据是共享的，数据节点指针会共享

AOF, RDB
当服务器运行在复制模式下的时候，主从服务器都能接受select服务，当某个键过期的时候，请求到达从服务器，这个时候从服务器是不知道该键已经过期的。当请求到达主服务器的时候
主服务器判断该键已经过期，会删除改建，并且同步消息给从服务器。

RDB AOF的持久化方式

RDB的备份数据方式是全面性的，SAVE的命令会导致主线程阻塞，BGSAVE的命令是在异步子进程中执行的，不会阻塞主进程。一般是server主进程定期执行BGSAVE命令。
AOF的同步方式会比RDB的方式好一些，是以命令追加的方式写入文件，并且会合并命令。由于CPU和磁盘响应的巨大差异有一些参数(appendfsync)可以控制AOF的同步方式，
appendfsync的取值有三种, 三种方式下各有缺点，不同的参数对Redis的性能有很大影响。
其中AOF的文件追加里面的命令重写比较有意思，client -> server -> append AOF & override AOF(child process). 当override AOF完成的时候会通知parent process
